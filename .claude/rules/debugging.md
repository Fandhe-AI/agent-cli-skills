# 根本原因デバッグ規約

根本原因の調査を終える前に修正コードを書かない。症状への当て推量的な修正は禁止。4フェーズを順に踏んだ上で修正を開始する。

## 適用対象

| 区分 | 対象 |
|------|------|
| スキル | implement-issue, implement-issue-tree |
| Agent | sub-investigator、実装を行う全 Agent |

## 4フェーズ

### Phase 1: 根本原因調査

- エラーメッセージ・スタックトレースを精読する（要約せず全文を確認）
- 最小再現手順を特定する
- 直近の変更差分（`git diff`・`git log -p`）と症状の関連を確認する
- データフロー・呼び出し経路を遡及してエラー発生箇所を特定する

### Phase 2: パターン分析

- 同じコードベース内で正常動作している類似実装を探す
- 失敗ケースと成功ケースの構造的差分を列挙する
- 共通パターン（命名・引数・型・依存関係）からの逸脱を確認する

### Phase 3: 仮説と検証

- 根本原因の仮説を**文字で記述**する（「〜が原因と考える。理由: 〜」）
- 一度に1変数のみ変更して検証する（複数箇所を同時修正しない）
- 仮説が外れた場合は Phase 1 に戻る（推測の積み重ねで進まない）

### Phase 4: 修正

- 可能な場合は失敗テストを先行して作成する（修正前にテストが RED になることを確認）
- 根本原因に対する単一集中修正を行う（関連して気になる箇所を同時修正しない）
- 修正後は `./verification.md` の5段階ゲートで完了を確認する

## エスカレーション則

同一箇所で **3回修正に失敗した**場合、アーキテクチャ問題または仕様の誤解と判断する。追加の修正試行は行わず、以下を明記してユーザー対話へ切り替える:

- 試行した修正内容と結果（各回）
- 現在の仮説と行き詰まりの理由
- ユーザーに確認が必要な事項（設計方針・外部依存・仕様の曖昧さ等）

## 禁止事項

- 根本原因が不明なまま「とりあえず修正」を試みること
- 複数の仮説に対する変更を同時に適用すること（何が効いたか判別不能になる）
- エラーを抑制・隠蔽して「解決済み」と報告すること
- 同一バグへの修正を4回以上試みること（エスカレーション則に従う）

## 関連ルール

- `./verification.md` — 修正後の完了ゲート規約
- `./security.md` — セキュリティ問題の調査・エスカレーション
- `./conventional-commits.md` — 修正コミットの記述規約（`fix` type）
