# コードコメントスタイル規約

コメントの中心思想は次の2点:

1. **役割の境界を書く** — パッケージ・サービスの視点から、対象（関数・型・モジュール）がシステム全体の中で何を担うかを書く。単に「何をするか」ではなく「なぜここにあり、どの境界を担うか」を明示する。
2. **読み手は渡された情報からしか判断できない** — 後続の読み手（人間および AI エージェント）は、他のファイル・他のサービス・呼び出し元/呼び出し先を自力で辿れるとは限らない。離れた場所にある前提・制約・契約をコメントで「その場」に持ち込む。

## 何を書くか

| 観点 | 書くべき内容 |
|------|-------------|
| 役割・責務の境界 | 「このモジュールは〜サービスの〜境界を担う」「〜パッケージの公開インターフェースとして機能する」 |
| 呼び出し元の文脈 | どのレイヤー・どのサービスから呼ばれるか。呼び出し元が前提とする状態・権限・契約 |
| 呼び出し先との契約 | 何を保証して返すか。エラー・例外の条件とその意味 |
| 他ファイル・他サービスとの依存 | 参照している外部 API・設定・共有状態のうち、読み手がファイルを跨がないと見つけられないもの |
| 非自明な制約・前提・不変条件 | なぜその実装になっているか（why）。背景・経緯・仕様上の制限 |

## 何を書かないか

- コードを読めば自明な逐語的説明（what の単純な言い換え）
- 実装と乖離して陳腐化しやすい重複情報（引数の型など、シグネチャに書いてある内容を繰り返すだけのもの）
- 一時的なメモや TODO は残しても良いが、解決済みの経緯のみを残したコメントは削除する

## 書き方の指針

### ドキュメンテーションコメント（JSDoc / docstring / rustdoc 等）

言語の慣習に従ったドキュメンテーションコメント形式を使い、対象の役割を1〜2行で要約する。

- 先頭の要約行に「役割・責務」を書く。「〜する」ではなく「〜の境界を担う」「〜を保証する」視点を含める
- 呼び出し元・呼び出し先の文脈が読み手に伝わらない場合は、その旨を本文に追記する
- 他ファイル・他サービスに依存する不変条件は、依存先を明示した上で記述する

### インラインコメント

- why（なぜその実装か）を書く。what はコードが示している
- 制約・背景・仕様上の都合は、該当行またはブロックの直前に書く
- 参照すべき外部情報（Issue 番号・仕様書 URL・設計ドキュメント）は積極的に記載する

## 良い例 / 悪い例

### 悪い例 — what の逐語的な言い換え

```
/**
 * ユーザーIDを受け取り、ユーザー情報を返す。
 * @param userId ユーザーID
 * @returns ユーザー情報
 */
function getUser(userId) { ... }
```

シグネチャを読めば分かる内容しか書かれていない。呼び出し元・呼び出し先の文脈も、境界の役割も伝わらない。

### 良い例 — 役割と他所からの観点を含む

```
/**
 * 認証レイヤーの公開インターフェース。API ハンドラーから呼ばれ、
 * セッション検証済みの呼び出しのみを前提とする（未認証は上流ミドルウェアで遮断）。
 *
 * UserRepository に委譲し、DB から取得した値を返す。
 * 存在しない場合は null を返す（例外は投げない）——
 * 呼び出し元は null チェックを必ず行うこと。
 *
 * 注: soft delete されたユーザーも null として扱う（仕様: issue #142）。
 */
function getUser(userId) { ... }
```

システム内の位置（認証レイヤー）、呼び出し元の前提（セッション検証済み）、返値の契約（null の意味）、非自明な仕様（soft delete の扱い）が「その場」で読める。

---

### 悪い例 — モジュールの役割が不明

```
// 注文処理
module OrderService { ... }
```

### 良い例 — サービス境界と依存を明示

```
/**
 * 注文ドメインの境界サービス。EC フロントエンドおよびバッチワーカーから呼ばれる。
 *
 * 在庫サービス（inventory-service）との同期的依存があり、
 * inventory-service が応答しない場合は注文確定を行わない（フェイルファスト）。
 * 非同期通知は notification-service へイベントを発行（fire-and-forget）。
 *
 * 外部 API（決済ゲートウェイ）の冪等キーは orderId を使用する——
 * 同一 orderId での再試行は安全（決済ゲートウェイ仕様 v3.2 §4.1）。
 */
module OrderService { ... }
```

---

### 悪い例 — なぜかが分からないインラインコメント

```
// リトライを3回に制限
const MAX_RETRY = 3;
```

### 良い例 — why と背景を書く

```
// 外部決済 API のレート制限（60 req/min）に従い3回に制限。
// 4回以上試みると一時 BAN される（2024年3月の障害対応で確認、issue #87）。
const MAX_RETRY = 3;
```

## 注意事項

- AI エージェント（Claude 等）がコードを読む際も、コメントが主要な文脈源となる。「他ファイルを参照すれば分かる」は通用しないと想定して書く
- コメントは実装変更時に必ず同期する。乖離したコメントは正確なコメントより有害
- セキュリティ上の制約・認証・権限の前提は特に丁寧に記述する（詳細は `./security.md`）
- コメントの言語は日本語を基本とする（詳細は `./japanese-style.md`）
