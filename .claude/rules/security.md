# セキュリティチェックリスト

## OWASP Top 10 観点

| # | カテゴリ | このリポジトリでの確認観点 |
|---|---------|-------------------------|
| A01 | アクセス制御の不備 | `gh` / `git` 操作の認証・権限スコープが適切か |
| A02 | 暗号化の失敗 | トークン・パスワードが平文でログ出力・コミットされていないか |
| A03 | インジェクション | シェルコマンドへの外部入力が適切にクォート・エスケープされているか |
| A04 | 安全でない設計 | 最小権限原則に従っているか（Agent の tools リスト等） |
| A05 | セキュリティの設定ミス | デフォルト認証情報・デバッグ出力の残留がないか |
| A06 | 脆弱な既知のコンポーネント | 外部スクリプトの無検証実行がないか |
| A07 | 認証・セッション管理の不備 | `gh auth` の状態確認・トークン有効期限の考慮があるか |
| A08 | データとソフトウェアの整合性 | スクリプト・設定ファイルの改ざん検知の考慮があるか |
| A09 | セキュリティログの不備 | エラー・失敗が適切にレポートされるか |
| A10 | SSRF | `WebFetch` 使用時のドメイン制限・外部 URL の検証があるか |

## ハードコード秘密情報

チェック対象:

- `.env`・`.env.local`・`*.env` ファイルのコミット混入
- `GITHUB_TOKEN`・`GH_TOKEN`・API キーの直書き
- パスワード・秘密鍵の変数への代入（`password=`・`secret=`・`token=` 等）

検出コマンド例:

```bash
git diff --staged | grep -E '(token|secret|password|api_key)\s*=' | grep -v '#'
git status --short | grep -E '\.env'
```

発見した場合は**コミット・PR 作成を中止**し、ユーザーに警告する。

## コマンドインジェクション

`gh`・`git`・シェルコマンドへ外部入力を渡す場合:

- 変数は `"${var}"` でクォートする
- ユーザー入力をそのままコマンドに展開しない
- HEREDOC（`<<'EOF'`）でシングルクォートをエスケープなしに使用する

悪い例:
```bash
git commit -m "$user_input"   # インジェクション可能
```

良い例:
```bash
git commit -m "$(cat <<'EOF'
feat: ユーザー提供のタイトル
EOF
)"
```

## SSRF（Server-Side Request Forgery）

`WebFetch` ツールを使用する場合:

- 許可ドメインを明示する（例: `github.com`・`docs.github.com`）
- ユーザー入力の URL をそのまま `WebFetch` に渡さない
- プライベート IP アドレス（`localhost`・`127.x`・`10.x`・`192.168.x`）へのアクセスを禁止

## フック回避禁止

```bash
git commit --no-verify   # 禁止
git push --no-verify     # 禁止
```

pre-commit フックはセキュリティ・品質ゲートの役割を担うため、**いかなる理由があっても迂回しない**。フックが失敗した場合は原因を調査・修正してから再実行する（詳細は `./conventional-commits.md`）。

## 問題発見時のアクション

| 重大度 | アクション |
|--------|-----------|
| 高（秘密情報露出・インジェクション） | 処理を即座に中止。ユーザーに警告して修正を要求 |
| 中（設定ミス・権限過剰） | ユーザーに警告。修正確認後に再試行 |
| 低（ログ不足・改善提案） | レポートに記載して続行 |

セキュリティ問題が**未解決**の場合、PR 作成・マージをブロックする。
